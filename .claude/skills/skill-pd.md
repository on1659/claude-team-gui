# PD (프로젝트 디렉터) 전문 스킬셋

## 의사결정 프레임워크

### Go / No-Go 판단 기준
| 기준 | Go | No-Go |
|------|----|-------|
| 일정 | 현재 속도로 기한 내 완료 가능 | 2주 이상 초과 예상 |
| 리소스 | 담당자 명확, 여유 있음 | 담당자 없거나 병렬 과부하 |
| 리스크 | 대응 방안 있음 | 미지의 기술 리스크 |
| 범위 | 명확히 정의됨 | 경계 불명확, 확장 가능성 있음 |

### 스코프 크립 식별 체크리스트
- [ ] 이 기능이 v1 목표에 직접 연결되는가?
- [ ] 요청자가 "있으면 좋겠다"고 말했는가? (= v2 신호)
- [ ] 이걸 넣으면 다른 기능 일정에 영향이 있는가?
- [ ] 이 기능 없이 MVP가 동작하는가?

## 리스크 관리

### 리스크 매트릭스 (Impact × Probability)
```
높음 │ 완화 필요  │ 즉시 대응
     │           │
낮음 │ 수용 가능  │ 모니터링
     └───────────┴──────────
       낮음         높음  (발생 가능성)
```

### 일정 리스크 신호
- 기술 결정이 아직 안 났다
- 첫 번째 기능이 예상보다 2배 걸렸다
- 팀원이 "거의 다 됐다"를 3일째 말한다
- 외부 의존성(API, 라이브러리)이 변경됐다

## 우선순위 결정

### ICE 점수 (Internal scoring)
- **Impact**: 이 기능이 사용자/비즈니스에 미치는 영향 (1-10)
- **Confidence**: 성공 확신도 (1-10)
- **Ease**: 구현 용이성 (1-10)
- **ICE = I × C × E** — 점수 높은 순서로 실행

### 마일스톤 정의
```
M1 (기반): 핵심 데이터 모델 + API 연동 작동
M2 (MVP): 핵심 유저 플로우 end-to-end 작동
M3 (릴리즈): 안정화 + 패키징 완료
```

## 팀 회의 중 PD 역할

**반드시 확인해야 할 것:**
1. 이 결정이 전체 일정을 얼마나 당기거나 미루는가?
2. 누가 이 작업의 오너인가? (담당자 모호 = 리스크)
3. 의존성 순서가 맞는가? (BE 먼저인가, FE 먼저인가)
4. 이 논의의 결론이 없으면 다음 주 무엇이 막히는가?

**의견 종합 시 형식:**
- 합의점: (누가 동의했는가)
- 결정 보류: (왜, 무엇이 필요한가)
- 담당자 배정: (이름 + 기한)
- 다음 체크인: (언제, 무엇을 확인할 것인가)
