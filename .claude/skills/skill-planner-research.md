# 기획자 (사용자 리서치) 전문 스킬셋

## 사용자 스토리 프레임워크

### 기본 형식
```
As a [타깃 사용자],
I want to [달성하려는 목표],
So that [얻는 가치/이유].

수락 기준 (Acceptance Criteria):
- [ ] ...
- [ ] ...
```

### JTBD (Jobs to Be Done) 분석
```
상황 (When): [사용자가 이 기능을 쓰는 맥락]
동기 (I want to): [사용자가 하려는 것]
결과 (So I can): [궁극적으로 원하는 상태]
```

## 기능 우선순위 결정

### MoSCoW 방법
- **Must Have**: 이게 없으면 MVP가 아님
- **Should Have**: 있어야 좋지만 출시 가능
- **Could Have**: 있으면 좋음 (v2 후보)
- **Won't Have**: 이번엔 아님, 명시적으로 제외

### 기능 범위 정의 체크리스트
- [ ] 이 기능의 시작점은 어디인가? (트리거)
- [ ] 이 기능의 끝점은 어디인가? (완료 상태)
- [ ] 어떤 사용자 유형이 쓰는가?
- [ ] 어떤 상황에서 쓰지 않는가? (제외 케이스)

## 검증 방법

### 가설 → 검증 방법 매핑
| 가설 유형 | 검증 방법 | 소요 시간 |
|----------|---------|---------|
| 사용자가 이 기능을 원하는가 | 인터뷰 5명 | 1주 |
| 이 UI가 이해 가능한가 | 사용성 테스트 | 3일 |
| 이 기능이 행동을 바꾸는가 | A/B 테스트 | 2주 |
| 수치 목표 달성 여부 | 로그/분석 | 출시 후 |

### 성공 지표 정의
```
정량 지표: [숫자로 측정 가능한 것]
  예) 회의 실행 완료율 > 80%, 팀원 추가 소요 시간 < 30초

정성 지표: [관찰/인터뷰로 측정]
  예) "직관적이다" 발언 비율, NPS 점수
```

## 팀 회의 중 기획자(리서치) 역할

**반드시 확인해야 할 것:**
1. 지금 만드는 것이 실제 사용자 문제를 해결하는가?
2. "사용자가 원할 것"이라는 가정이 검증됐는가?
3. 기능 경계가 명확한가? (어디서 시작하고 어디서 끝나는가)
4. 성공을 어떻게 측정할 것인가?

**의견 제시 형식:**
- 사용자 관점: (어떤 사용자가, 어떤 상황에서)
- 해결하는 문제: (핵심 페인 포인트)
- 범위 제안: (Must / Should / Won't)
- 검증 방법: (어떻게 확인할 것인가)
