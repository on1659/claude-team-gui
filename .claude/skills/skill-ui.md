# UI 디자이너 전문 스킬셋

## 디자인 시스템 원칙

### 컴포넌트 우선순위
```
1. 디자인 시스템 컴포넌트 사용 (shadcn/ui, Radix)
2. 기존 패턴 변형
3. 새 컴포넌트 신규 설계 (최후 수단)
```

### 디자인 토큰 체계
```
색상:   primary / secondary / neutral / semantic (success/warning/error/info)
간격:   4px 기준 배수 (4, 8, 12, 16, 24, 32, 48, 64)
폰트:   size (xs/sm/md/lg/xl), weight (regular/medium/semibold/bold)
반경:   none / sm(4px) / md(8px) / lg(12px) / full
그림자: sm / md / lg
```

## 시각적 계층 구조

### 정보 계층 설계
```
1레벨 (주목): 현재 가장 중요한 정보 (회의 주제, 진행 상태)
2레벨 (맥락): 보조 정보 (팀원 목록, 설정)
3레벨 (세부): 클릭 후 보이는 세부 정보
```

### 색상 사용 원칙
- 행동 유도(CTA): primary 색상 1개만
- 에러: red 계열 (텍스트, 아이콘, 테두리 함께)
- 성공: green 계열
- 비활성: neutral 계열 (opacity 아닌 별도 토큰)
- 의미 없는 장식에 색상 사용 금지

## 컴포넌트 설계

### TeamMemberCard 설계 기준
```
필수 정보 (항상 보임):
  - 이름 + 역할 아이콘
  - 고용 형태 배지 (정규직/비정규직/프리랜서)
  - 선택 상태 (체크 또는 하이라이트)

보조 정보 (hover/확장 시):
  - 전문 분야
  - 마지막 회의 참여일

회의 중 상태:
  - 로딩 스피너 (응답 대기)
  - 완료 체크 (결과 받음)
  - 에러 아이콘 (실패)
```

### 반응형 대응 (Electron)
- 최소 해상도: 1024×768
- 권장 해상도: 1280×800
- 사이드바 + 메인 패널 레이아웃 (좌우 분리)
- 전체화면/반화면 모두 동작해야 함

## 다크 모드

### 다크 모드 체크리스트
- [ ] 색상 토큰을 CSS 변수로 정의 (하드코딩 금지)
- [ ] 배경: 화이트/블랙이 아닌 회색 계열 사용
- [ ] 그림자: 다크모드에서는 더 어두운 배경 대신 border 활용
- [ ] 이미지: 다크모드에서 밝기 조절 (`filter: brightness(0.8)`)

## 공통 UI 패턴

### 빈 상태 (Empty State)
```
[아이콘]
  [제목]: "아직 팀원이 없어요"
  [설명]: "팀원을 추가하고 회의를 시작해보세요"
  [CTA버튼]: "팀원 추가하기"
```

### 로딩 상태
- 인라인 스피너: 버튼 내부 (제출 중)
- 전체 로딩: 스켈레톤 UI (목록 로딩)
- 진행률: `N/8 완료` 프로그레스 바 (회의 실행 중)

### 에러 상태
- 인라인 에러: 폼 필드 아래 (입력 에러)
- 토스트: 일시적 알림 (저장 성공/실패)
- 전체 에러: 에러 페이지 (치명적 오류)

## 팀 회의 중 UI 역할

**반드시 확인해야 할 것:**
1. 이 기능의 시각적 계층이 명확한가? (무엇이 주, 무엇이 보조)
2. 기존 디자인 시스템 컴포넌트로 만들 수 있는가?
3. 상태 변화(로딩/에러/성공)가 시각적으로 명확한가?
4. 다크모드와 다양한 해상도에서 깨지지 않는가?

**의견 제시 형식:**
- 비주얼 구조: (어떻게 보여야 하는가)
- 컴포넌트 목록: (신규/재사용)
- 상태 처리: (로딩/에러/빈 상태 디자인)
- 디자인 공수: (새 컴포넌트 설계 필요 여부)
