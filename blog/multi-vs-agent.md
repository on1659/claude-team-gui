# Claude 멀티에이전트 vs 역할극: 진짜 차이는 뭘까?

> Claude Code에서 여러 역할의 의견을 모으는 두 가지 방법, 어떻게 다를까?

---

## 들어가며

Claude로 팀 회의를 시뮬레이션하는 방법은 크게 두 가지가 있다.

- **역할극 방식** (`/meeting-multi`): 하나의 Claude가 여러 역할을 순서대로 연기
- **진짜 멀티에이전트** (`/meeting-agent`): 여러 Claude 인스턴스가 독립적으로 병렬 실행

이름만 보면 비슷해 보이지만, 내부 동작 방식은 완전히 다르다.

---

## 역할극 방식: 1인 다역 배우

### 동작 원리

```
[Claude 1개, 단일 컨텍스트]

→ "저는 기획자입니다. 이 기능은 사용자 니즈에 부합합니다."
→ "저는 개발자입니다. 기술적으로 구현 가능합니다." ← 기획자 말을 읽고 답변
→ "저는 QA입니다. 다음 리스크가 있습니다."         ← 기획자+개발자 말을 읽고 답변
```

모든 역할이 **같은 메모리**를 공유한다. 나중에 말하는 역할일수록 앞선 발언을 전부 읽은 상태에서 답한다.

### 특징

| 항목 | 내용 |
|------|------|
| Claude 인스턴스 | 1개 |
| 실행 방식 | 순차 (하나씩) |
| 컨텍스트 | 공유 (누적) |
| 독립성 | 없음 — 앞 발언에 영향 받음 |
| 속도 | 빠름 |

### 한계

역할극이기 때문에 구조적으로 **편향**이 발생한다. 기획자가 강한 의견을 먼저 내면, 뒤에 등장하는 개발자와 QA는 무의식적으로 그 의견에 동조하거나 반박하는 방향으로 흐를 수 있다. 실제 팀 회의에서 먼저 발언한 사람이 분위기를 좌우하는 것과 같은 현상이다.

---

## 진짜 멀티에이전트: 실제 팀

### 동작 원리

```
[오케스트레이터 Claude]
        ↓ Agent 도구로 동시 실행

[기획자①]  [기획자②]  [개발자①]  [개발자②]  [QA]  [UI]  [UX]  [PD]
    ↓           ↓           ↓           ↓        ↓     ↓     ↓    ↓
 주제만 알고  주제만 알고  주제만 알고  주제만 알고  ...
 독립 분석    독립 분석    독립 분석    독립 분석

        ↓ 모든 결과 수집
[오케스트레이터 Claude]
        ↓
    최종 종합
```

각 에이전트는 **자신의 역할과 주제만 알고 분석**한다. 다른 역할이 뭐라고 했는지 전혀 모른다.

### 특징

| 항목 | 내용 |
|------|------|
| Claude 인스턴스 | 역할 수만큼 (8명이면 8개 + 오케스트레이터) |
| 실행 방식 | 병렬 (동시에) |
| 컨텍스트 | 독립 (각자 별도) |
| 독립성 | 완전 — 서로 영향 없음 |
| 속도 | 병렬 실행으로 비슷하거나 빠를 수 있음 |

### 장점

편향이 없다. 기획자의 발언이 개발자의 판단에 영향을 미치지 않는다. 각 전문가가 순수하게 자신의 관점에서만 분석하기 때문에 **더 다양하고 날카로운 의견**이 나올 수 있다.

## 토큰 사용량 비교

### 역할극 방식의 토큰 누적 구조

역할극은 컨텍스트가 계속 쌓이는 구조다. 역할이 추가될수록 이전 발언을 모두 읽어야 하므로 입력 토큰이 급격히 늘어난다.

```
역할 1: 프롬프트(500) + 주제(50)                          = 550 input
역할 2: 프롬프트(500) + 주제(50) + 역할1 결과(300)         = 850 input
역할 3: 프롬프트(500) + 주제(50) + 역할1+2 결과(600)       = 1,150 input
역할 4: 프롬프트(500) + 주제(50) + 역할1~3 결과(900)       = 1,450 input
...
역할 8: 프롬프트(500) + 주제(50) + 역할1~7 결과(2,100)     = 2,650 input
```

8명 기준 추정:
- **총 input: ~10,000 토큰**
- **총 output: ~2,400 토큰**
- **합계: ~12,400 토큰**

### 멀티에이전트 방식의 토큰 구조

각 에이전트는 독립 컨텍스트에서 실행되므로 누적이 없다.

```
에이전트 8개 각각:
  프롬프트(500) + 역할 지시(200) + 주제(50) = 750 input
  8명 × 750 = 6,000 input
  8명 × 300 = 2,400 output

오케스트레이터 종합:
  8명 결과 수신: 300 × 8 = 2,400 input
  종합 output: 500
```

8명 기준 추정:
- **총 input: ~8,400 토큰**
- **총 output: ~2,900 토큰**
- **합계: ~11,300 토큰**

### 요약

| | 역할극 (multi) | 멀티에이전트 (agent) |
|---|---|---|
| 예상 토큰 | ~12,400 | ~11,300 |
| 차이 | — | 약 10% 절약 |
| 인원 증가 시 | 급격히 증가 (누적) | 선형 증가 |

**인원이 많아질수록 멀티에이전트가 훨씬 유리하다.** 역할극은 역할 수의 제곱에 가깝게 토큰이 늘지만, 멀티에이전트는 단순 곱셈이기 때문이다.

## 에이전트의 수명

한 가지 중요한 개념이 있다. 에이전트는 **일회용**이다.

```
에이전트 실행 → 작업 완료 → 결과 반환 → 소멸
```

마치 계약직 프리랜서처럼, 일을 맡기면 결과물을 주고 사라진다. 다음 호출 때는 이전 대화를 기억하지 못한다.

단, `resume` 파라미터로 이전 에이전트를 재소환할 수 있다:

```
1차 실행 → agent_id: "abc123" 반환됨
                    ↓
resume: "abc123" → 이전 컨텍스트 그대로 이어서 실행
```

이 기능을 활용하면 에이전트와 이어서 대화하거나, 추가 작업을 지시할 수 있다. 다만 세션이 끊기면 resume도 불가능하므로 장기적인 팀원 유지는 현재 Claude Code로는 어렵다.

## 언제 어떤 방식을 쓸까?

| 상황 | 추천 방식 | 이유 |
|------|-----------|------|
| 빠르게 다양한 관점 원할 때 | 역할극 (multi) | 설정 없이 빠름 |
| 서로 편향 없는 독립 의견 필요 | 멀티에이전트 (agent) | 컨텍스트 완전 독립 |
| 역할이 5명 이상 | 멀티에이전트 (agent) | 토큰 효율 우위 |
| 빠른 브레인스토밍 | 역할극 (multi) | 컨텍스트 공유로 자연스러운 흐름 |
| 중요한 아키텍처 결정 | 멀티에이전트 (agent) | 독립성이 결과 품질에 영향 |

---

## 마치며

두 방식 모두 Claude의 강력한 기능을 활용하는 방법이지만, 목적이 다르다.

- **역할극**은 빠르고 자연스러운 흐름이 필요할 때 유용하다
- **멀티에이전트**는 독립적이고 편향 없는 분석이 필요할 때 진가를 발휘한다

토큰 효율 측면에서도 큰 차이는 없지만, 인원이 늘어날수록 멀티에이전트가 유리하다. 그리고 무엇보다, 에이전트는 서로의 발언을 모른다는 것 — 이것이 가장 큰 차이이자 핵심이다.
