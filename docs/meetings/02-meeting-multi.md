# `/meeting-multi` — 역할극 회의 (8인)

---

## 개요

Claude 1개가 8개 역할을 **순서대로** 연기하는 방식. 앞 발언이 뒤 역할에 영향을 미쳐 자연스러운 토론 흐름이 만들어진다. 단, 구조적으로 편향이 발생한다.

---

## 동작 방식

```
[Claude 1개, 단일 컨텍스트]

→ 기획자① 발언  (주제만 앎)
→ 기획자② 발언  (주제 + 기획자① 발언 읽음)
→ 개발자① 발언  (주제 + 기획자①② 발언 읽음)
→ 개발자② 발언  (주제 + 기획자①② + 개발자① 읽음)
→ QA 발언       (모든 앞 발언 읽음)
→ UI 발언       (...)
→ UX 발언       (...)
→ PD 발언       (모든 발언 읽음)
→ 종합
```

### 편향 발생 구조

첫 번째 발언자가 강한 주장을 하면, 뒤 역할들은 그 주장에 동조하거나 반박하는 방향으로 흘러간다. 실제 회의에서 첫 발언자가 분위기를 좌우하는 것과 같은 현상.

## 출력 구조 예시

> 주제: "모바일 앱에 다크모드 추가"

```
---
**기획자①**
사용자 리서치 결과 야간 사용자의 67%가 다크모드를 원한다.
OLED 기기 배터리 절약 효과도 검증됨.
이번 분기 내 출시 필요.

---
**기획자②**
현우 의견에 동의. 경쟁 앱 3사 모두 이미 지원 중.       ← 기획자① 발언 참조
KPI: 야간 세션 시간 20% 증가 목표로 설정.

---
**개발자①**
구현 가능. React Native useColorScheme 활용.           ← 기획자들 방향 인식 후
기획에서 말한 이번 분기는 가능하나 기존 컴포넌트 전면 수정 필요.
테마 토큰 시스템 없으면 공수 2배.

---
**개발자②**
태준 말대로 테마 시스템 먼저 잡아야 한다.              ← 개발자① 발언 참조
상태 관리에 theme context 추가, Async Storage로 설정 저장.

---
**QA**
기획·개발 논의 봤을 때 테마 전환 시 플리커 케이스 중요.  ← 모든 앞 발언 참조
라이트/다크 전환 + 기기 시스템 설정 연동 테스트 필수.
출시 전 최소 3개 기기 검증 필요.

---
...이하 UI, UX, PD 발언...

## 🏁 팀 회의 결과: 다크모드 추가

### 공통 합의점
- 구현 가치 있음, 이번 분기 가능
- 테마 시스템 선행 구축 필요

### 주요 쟁점
- 테마 시스템 구축 공수 포함 여부 (일정 영향)

### 권장 다음 액션
1. 테마 토큰 시스템 설계 (개발자①② 담당)
2. QA 테스트 기기 목록 확정
3. KPI 측정 방법 구체화 (기획자②)
```

---

## 특징

| 항목 | 내용 |
|------|------|
| 토큰 (8인) | ~19,200 |
| 속도 | 순차 실행 — 느림 |
| 편향 | **있음** — 앞 발언에 영향받음 |
| 인원 증가 시 | 토큰 급증 (누적 구조) |
| 출력 느낌 | 실제 토론처럼 자연스러운 흐름 |

---

## 다른 방식과의 차이

- `/meeting` 대비: 인원 많고 깊이 있음. 토론 흐름 있음.
- `/meeting-agent` 대비: 같은 인원이지만 컨텍스트 공유 → 편향 발생, 토큰 더 많음.
- `/meeting-team` 대비: 팀원 정체성(이름/성격) 없음. 역할만.

---

## 적합한 상황

- 토론이 이어지는 흐름이 중요할 때
- 앞 역할 의견에 반응하는 다이나믹이 필요할 때
- 편향보다 자연스러운 흐름이 우선일 때
- 인원이 4명 이하일 때 (5명 이상은 `/meeting-agent`가 토큰 효율 우위)
