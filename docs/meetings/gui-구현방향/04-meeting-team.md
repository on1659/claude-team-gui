# `/meeting-team` 실행 결과

**주제**: Claude Code 팀원 GUI 프로젝트 구현 방향 결정
**방식**: 팀원 프로필 주입 후 8개 독립 인스턴스 병렬 실행
**특징**: 이름·성격·말투·판단 기준 유지, 서로의 발언 모름

---

## 사용한 프롬프트 (공통 구조)

```
당신은 [이름]입니다. 다음은 당신의 프로필입니다:
[해당 팀원 프로필 전체]

회의 주제: Claude Code 팀원 시스템(회의 방식 3종: multi/agent/team)을
GUI 앱으로 구현하는 방향 결정

미결 사항:
- 프레임워크: Electron vs Tauri vs 웹앱
- 아키텍처: Anthropic API 직접 호출 vs Claude Code CLI 래핑
- 데이터: profiles.md (Markdown) 유지 vs JSON 전환

위 프로필의 성격과 말투 그대로 다음을 작성하세요:
1. 핵심 의견 / 2. 우려사항 / 3. 제안하는 방향 / 4. 다른 팀원에게 묻고 싶은 것
다른 팀원의 의견은 모르는 상태입니다.
```

### 지민 (PD)

**핵심 의견**
솔직하게 말할게요. 지금 세 가지 미결 사항은 따로따로 결정할 수 있는 게 아닙니다. 프레임워크 선택이 아키텍처를 제약하고, 아키텍처가 데이터 복잡도를 결정해요. PD 입장에서 가장 중요한 건 "이상적인 스택"이 아니라 **"완성 가능한 스택"** 입니다.

**우려사항**
- Tauri: Rust 역량 없으면 학습 곡선이 일정을 그냥 잡아먹음
- API 직접 호출: 팀원 관리·컨텍스트 주입·세션 핸들링을 전부 직접 구현 → 스코프가 조용히 커지는 패턴
- 회의 방식 3종 동시 개발 + 데이터 포맷 전환 동시에 하면 첫 동작 버전이 늦어짐

**제안**
Electron (팀이 JS/TS 쓸 수 있으면 검증된 선택). CLI 래핑 + 추상화 레이어 (나중에 API 직접 전환 가능하게). profiles.md 1단계 유지 + 파싱 모듈 분리. **MVP 먼저, 확장은 나중에.**

**다른 팀원에게**
> 개발자에게: CLI 래핑 방식에서 회의 방식 3종 중 가장 복잡한 게 뭔가요? 그리고 첫 동작 버전 기준으로 몇 주 안에 나올 수 있나요? 막연한 "할 수 있다"가 아니라 숫자가 필요합니다.

### 현우 (기획자①)

**핵심 의견**
지금 논의가 "어떻게 만들까"에 집중돼 있고 **"누가 왜 쓰는가"가 빠져 있어요.** 타깃 사용자는 두 그룹:
- A그룹: 개발자이지만 CLI 설정이 번거롭고 반복 사용하고 싶은 사람
- B그룹: 비개발자로 CLI 자체가 진입 장벽인 사람

이 두 그룹이 필요한 게 완전히 달라요. 타깃 전제가 정해진 후에 기술 선택을 해야 의미가 있어요.

**우려사항**
- 검증 없는 기능 설계: 회의 방식 3종 차이를 사용자가 인지하고 선택하는지 확인된 적 있나요?
- CLI 래핑의 UX 함정: 오류 메시지·설치 의존성이 그대로 GUI에 노출 → 비개발자에게 치명적
- profiles.md 유지 시 GUI 편집 UX 불가

**제안**
타깃을 A그룹(개발자)으로 먼저 좁히고 MVP 작게. Electron + API 직접 + JSON 전환.

**다른 팀원에게**
> 개발자에게: CLI 래핑 방식에서 Claude Code 미설치 환경 에러를 GUI에서 얼마나 깔끔하게 숨길 수 있나요? 이게 비개발자 타깃 가능 여부의 분기점이에요.

### 소연 (기획자②)

**핵심 의견**
솔직히 말할게요. 지금 결정해야 할 건 기술 선택이 아니라 **"누구에게 팔 거냐"** 예요. 타깃이 명확하지 않으면 어떤 프레임워크를 골라도 의미 없어요. 세 가지 미결 사항은 전부 비즈니스 포지셔닝 문제예요.

**우려사항**
- 시장 타이밍: AI 협업 툴 시장이 폭발 중. 내부 완성도에 집중하다 늦어지면 경쟁 툴이 레퍼런스가 됨
- API 직접 호출 시 과금 구조 설계 지금부터 필요 (사용자 API 키 vs 마진 구조에 따라 수익 모델 달라짐)
- profiles.md 유지 시 팀 단위 관리·KPI 추적 불가 → 기술 부채로 돌아옴

**제안**
웹앱 우선 (설치 없이 온보딩 전환율 확보). API 직접 호출 (로드맵 독립성). JSON 즉시 전환.

**다른 팀원에게**
> 개발자에게: 웹앱으로 갔을 때 회의 방식 3종 지원 MVP 실제로 몇 주 안에 나와요? 출시 타이밍이 제일 중요한 변수라 구체적인 숫자가 필요해요.

### 태준 (개발자①·백엔드)

**핵심 의견**
솔직히 말하면, **아키텍처 선택이 프레임워크 선택보다 훨씬 중요**합니다. API 직접 호출은 제어권이 있지만 CLI 기능을 재구현해야 함. CLI 래핑은 IPC 설계 복잡. 데이터는 반드시 JSON — MD는 파싱 오버헤드·스키마 불안정성이 실무 허용 불가.

**우려사항**
- CLI 래핑: stdout 파싱이 버전에 종속, 버전 업데이트 시 파싱 깨짐
- API 직접: agent 방식 tool_use 루프 구현 난이도 높음
- API Key: Electron 렌더러 노출 위험
- profiles.md: 멀티 프로세스 동시 R/W 시 race condition
- 병렬 스트리밍 백프레셔 처리 미비 시 메모리 누수

**제안**
**Tauri** (Rust 백엔드가 API Key·파일I/O 격리). API 직접 호출. CLI 래핑 원한다면 중간 어댑터 레이어:
```
GUI → Tauri Command → MeetingAdapter
                         ├── ClaudeCliAdapter (현재)
                         └── AnthropicApiAdapter (추후)
```
JSON + SQLite 하이브리드. schema_version 필드 초기부터 포함.

**다른 팀원에게**
> UX/프론트에게: GUI에서 프로필 편집 기능을 초기 범위에 넣나요? 편집 UI가 있으면 JSON 전환은 협상 대상이 아닙니다.

### 미래 (개발자②·프론트엔드)

**핵심 의견**
솔직히 말하면, **프레임워크 선택이 UX 품질을 거의 결정**해요. 스트리밍이 얼마나 부드럽게 나오는지, 에이전트 여러 개 동시에 돌릴 때 UI가 버티는지 — 기능 명세보다 렌더링 파이프라인 설계가 먼저예요.

**우려사항**
- Electron: 번들 200MB+, 메모리 사용 높음. 3모드 탭 관리 시 메모리 압박
- Tauri: IPC 추가 공수, OS별 WebView 스타일 깨짐 리스크
- 웹앱: 배포 쉽지만 subprocess 관리·파일시스템 접근에 백엔드 서버 추가 필요 → 아키텍처 복잡화
- CLI 래핑 + 프론트: stdout 포맷 변경 시 프론트도 동시에 터지는 커플링

**제안**
**Electron** (VS Code·Cursor도 Electron, 번들 크기는 첫 릴리즈 수준에서 감수 가능). CLI 래핑 + 얇은 IPC 레이어 (기존 CLI 자산 활용). JSON 전환 강력 권고. 컴포넌트: 공통 ChatPanel + 모드별 ControlBar.

**다른 팀원에게**
> 백엔드에게: CLI 래핑 방식에서 multi 모드처럼 에이전트 여러 개 동시에 돌릴 때 프로세스 라이프사이클 관리를 백엔드가 책임지는 구조 가능한가요? 확정돼야 프론트 상태 관리 설계 시작할 수 있어요.

---
### 윤서 (QA)

**핵심 의견**
솔직히 말하면, 프레임워크·아키텍처 결정보다 **"어떻게 검증할 것인가"가 먼저 정의되어야** 합니다. 세 가지 선택지 모두 테스트 자동화 난이도가 완전히 달라요. 지금 결정이 나중에 QA가 수작업 지옥에 빠지는지 아닌지를 결정해요.

**우려사항**
- [HIGH] 회의 방식 3종 상태 전이 검증: CLI 래핑이면 프로세스 상태까지 추적해야 해 테스트 복잡도 폭증
- [HIGH] 비동기 스트리밍 엣지케이스: 네트워크 불안정·타임아웃·부분 수신 시나리오 전부 케이스
- [MEDIUM] profiles.md 파싱: 줄바꿈 하나에 프로필 깨짐, JSON이면 스키마 검증으로 즉시 catch
- [MEDIUM] 플랫폼 크로스 검증: Electron은 Win/Mac/Linux 전부, Tauri는 WebView 엔진까지

**제안**
**웹앱 (Next.js계열)** (Playwright E2E + Vitest 단위 + MSW API 모킹 파이프라인). API 직접 호출 (레이어별 독립 테스트·모킹 가능). JSON + Zod 스키마 검증. **릴리즈 게이팅 기준 지금 합의 필요.**

**다른 팀원에게**
> 개발팀에게: 회의 방식 3종의 상태 전이 조건과 실패 시 롤백 시나리오가 문서화되어 있나요? 개발 시작 전에 상태 다이어그램 수준이라도 뽑아줄 수 있나요?

### 다은 (UI 디자이너)

**핵심 의견**
솔직히 말하면, **프레임워크 선택보다 디자인 시스템 먼저 잡는 게 맞아요.** 어떤 프레임워크든 사용자가 보는 건 UI거든요. 근데 지금 논의에 "이 앱이 어떻게 생겨야 하는가"가 없잖아요. 다크모드 대응도 처음부터 CSS 변수 기반 테마 시스템이 아니면 나중에 엄청 힘들어요.

**우려사항**
- 컴포넌트 일관성: 3모드 각각 개발하다 보면 버튼 스타일 3가지·카드 레이아웃 4가지 상황 발생
- profiles.md 유지 시 팀원 카드 렌더링·역할별 태그·아바타 커스터마이징 모두 제약
- 다크모드 초기 미설계 시 나중에 전면 리작업

**제안**
**Tauri + React + shadcn/ui** (shadcn/ui가 디자인 시스템 시간 절감, Tailwind 테마 변수 관리 깔끔). JSON 전환 (역할·색상·아이콘·판단 기준을 필드로 분리해 UI 바인딩). 디자인 토큰 + 컴포넌트 시스템 먼저 정의 후 개발 시작.

**다른 팀원에게**
> 개발에게: multi/agent/team 세 가지 모드가 런타임에 동적으로 전환되나요, 앱 시작 시 선택인가요? 동적 전환이면 레이아웃 전환 애니메이션·상태 관리까지 처음부터 설계에 넣어야 해요.

### 승호 (UX 디자이너)

**핵심 의견**
저는 프레임워크 선택보다 **사용자가 세 가지 회의 모드를 얼마나 직관적으로 이해하고 전환할 수 있느냐**가 핵심이라고 봐요. multi·agent·team이 첫 화면에서 5초 안에 파악되지 않으면 기술 스택이 아무리 좋아도 의미 없어요.

**우려사항**
- 인지 부하: 모드 차이를 텍스트로만 설명하면 선택 장벽에서 이탈
- 동시 응답 흐름: "지금 누가 말하고 있는지", "어떤 에이전트가 대기 중인지" 실시간 피드백 없으면 불안감
- 오류·빈 상태 설계 후순위로 밀림 → 가장 큰 UX 구멍
- ARIA live region 초기 미설계 시 스크린 리더 대응 대규모 수정 필요

**제안**
웹앱 또는 Electron (접근성 컴포넌트는 검증된 웹 스택 필요). API 직접 호출 (스트리밍 응답 중 실시간 피드백 가능). JSON 전환 (역할·판단 기준 필드 UI 컴포넌트 바인딩). 핵심 UX 원칙: "지금 어떤 모드인지" 항상 명시 / 에이전트별 발언 중·대기·완료 상태 시각 구분 / 오류·빈 상태 가이드 메시지 1순위.

**다른 팀원에게**
> 개발팀에게: 에이전트 응답이 스트리밍으로 들어올 때 여러 에이전트가 동시에 스트리밍되는 경우도 있나요? 동시 스트리밍이면 화면 레이아웃과 포커스 관리 전략이 완전히 달라져요.

## 🏁 GUI 구현 방향 — 팀 회의 결과

### 팀원별 프레임워크 의견

| 팀원 | 제안 | 핵심 근거 |
|------|------|----------|
| 지민 (PD) | Electron | 완성 가능성, 팀 역량 |
| 현우 (기획①) | Electron | 개발자 타깃 먼저 |
| 소연 (기획②) | 웹앱 | 온보딩 전환율, 출시 속도 |
| 태준 (BE) | Tauri | 보안·성능, Rust 격리 |
| 미래 (FE) | Electron | 렌더링 파이프라인, 생태계 |
| 윤서 (QA) | 웹앱 | 테스트 자동화 파이프라인 |
| 다은 (UI) | Tauri | shadcn/ui, 디자인 시스템 |
| 승호 (UX) | 웹앱/Electron | 접근성 컴포넌트 |

### 주요 합의점 (전원 동의)
- **Anthropic API 직접 호출** — CLI 래핑 반대 압도적 (CLI 버전 종속·테스트 불가·UX 함정)
- **JSON 전환** — 반대 의견 없음 (UI 바인딩·테스트·확장성 전부 유리)

### 주요 충돌 지점
- **프레임워크**: Electron(3표) vs 웹앱(3표) vs Tauri(2표) — 완전한 삼파전
- **타깃 사용자**: 개발자 한정(현우·지민) vs 비개발자 포함(소연·승호) — 이것이 프레임워크 결정의 실제 기준
- **MVP 범위**: 1종 방식부터(지민·PD) vs 3종 동시(소연)

### 서로에게 던진 질문들 (다음 회의 아젠다)
- **개발자 → FE/UX**: 프로필 편집 UI가 초기 범위인가? (데이터 설계 선행 조건)
- **기획자 → 개발자**: CLI 미설치 환경 에러 핸들링 범위
- **기획자 → 개발자**: 웹앱 기준 MVP 몇 주 소요?
- **UX → 개발자**: 에이전트 동시 스트리밍 가능한가? (레이아웃 설계 분기점)

## meeting-agent와의 실제 차이

| 비교 항목 | meeting-agent | meeting-team |
|-----------|--------------|--------------|
| 발언 톤 | 역할 보고서 형식 | 실제 말투 ("솔직히 말할게요", "그거 아닌 것 같아요") |
| 질문 항목 | 없음 | 팀원별 1개씩 구체적 질문 → 다음 회의 아젠다 |
| 의견 근거 | 역할 관점 일반론 | 자기 업무 경험 기반 구체적 사례 |
| 충돌 지점 | 표 정리 수준 | 누가 누구와 어떤 이유로 충돌하는지 명시 |
| 프레임워크 투표 | 웹앱 4 / Electron 3 / Tauri 1 | Electron 3 / 웹앱 3 / Tauri 2 (다른 분포) |
