## 3. 방식별 상세 분석

### `/meeting` — 9,286 토큰 / 13점

**무엇을 잘했나**
- 3역할 3줄씩 — 가장 빠르게 방향 잡기
- "Electron + API 직접 + MD 유지"로 즉시 결론

**무엇이 부족했나**
- PD가 기획자+개발자 발언을 읽고 "이걸로 확정"이라고 단정 → **인위적 수렴**
- 3역할이라 QA·UX·UI 관점 완전 누락
- profiles.md 유지 결론 → 4종 중 유일하게 JSON 전환 안 함 (편향 영향)

**적합한 상황**: 혼자 5분 안에 빠른 방향 체크할 때

---

### `/meeting-multi` — 12,109 토큰 / 17점

**무엇을 잘했나**
- 8역할 커버. 자연스러운 토론 흐름
- QA가 "부분 실패(partial failure) retry 로직" 지적 → meeting에서 나오지 않은 구체적 이슈
- UX가 "3/8 완료 표시" UX 아이디어 제안
- 결론이 명확: Electron + API 직접 + MD→JSON 파싱

**무엇이 부족했나**
- 개발자①이 "Electron 확정"이라고 선언하자 이후 역할들이 그 전제 위에서 논의 → **편향 발생**
- meeting-agent에서는 웹앱 4표였는데 여기서는 Electron으로 수렴 → 첫 발언이 결론을 유도

**적합한 상황**: 토론 흐름을 기록으로 남기고 싶을 때, 토큰 아끼면서 8역할 원할 때

---

### `/meeting-agent` — 81,434 토큰 / 20점

**무엇을 잘했나**
- 편향 없는 진짜 분포 확인: 웹앱 4표 vs Electron 3표 vs Tauri 1표
- multi에서는 Electron으로 수렴됐는데, 독립 실행하니 웹앱 지지가 더 많다는 게 드러남 → **이게 실제 의견 분포**
- API 직접 호출·JSON 전환은 전원 동의 → 흔들리지 않는 합의점 확인
- 각 역할이 서로 모르는 상태라 더 날카로운 리스크 발굴

**무엇이 부족했나**
- 토큰 8.8배 — 비용 부담
- 다음 회의 아젠다가 될 "질문"이 없음
- 충돌 지점이 "표 정리" 수준 — 누가 왜 다른지 맥락 없음

**적합한 상황**: 중요한 기술 결정, 편향 없는 의견 분포가 필요할 때

---

### `/meeting-team` — 85,165 토큰 / 21점

**무엇을 잘했나**
- **상위 질문 발굴**: "타깃 사용자를 먼저 결정해야 프레임워크가 결정된다" — 다른 방식에서 나오지 않은 메타 인사이트
- 팀원별 **구체적 질문 6개** → 실제 다음 회의 아젠다가 됨
  - "프로필 편집 UI가 초기 범위인가?" (데이터 설계 선행 조건)
  - "에이전트 동시 스트리밍 가능한가?" (레이아웃 설계 분기점)
  - "회의 방식 상태 다이어그램 먼저 작성 가능한가?"
- 충돌 지점이 **사람 이름과 이유 포함** → 실제 팀 회의처럼 읽힘
- 태준(BE)이 어댑터 레이어 설계 코드까지 제안 → 구현 바로 써먹을 수 있는 수준

**무엇이 부족했나**
- 가장 비쌈 (9.2x)
- 프레임워크 삼파전(3-3-2)으로 결론 미결 — 오히려 현실적이긴 하지만

**적합한 상황**: 중요한 아키텍처 결정, 다음 회의 아젠다 만들 때, 팀원 캐릭터의 일관성이 중요할 때

## 4. 방식별 프레임워크 투표 비교

| 방식 | Electron | 웹앱 | Tauri | 비고 |
|------|:---:|:---:|:---:|------|
| `/meeting` | ✅ 확정 | — | — | 3역할 → 인위적 수렴 |
| `/meeting-multi` | ✅ 확정 | 언급 | — | 개발자① 발언 후 수렴 |
| `/meeting-agent` | 3표 | **4표** | 1표 | 편향 없는 실제 분포 |
| `/meeting-team` | **3표** | 3표 | 2표 | 삼파전, 타깃 사용자가 기준 |

> **결론**: meeting/multi는 편향으로 Electron 확정. agent/team은 실제로는 웹앱 지지가 많거나 팽팽하다. "Electron이 당연하다"는 인상은 편향의 결과였다.

## 5. 전 방식 공통 합의 (변하지 않는 결정)

4가지 방식 모두에서 반대 없이 동의한 항목:

| 항목 | 결정 | 근거 |
|------|------|------|
| **아키텍처** | Anthropic API 직접 호출 | CLI 래핑: 버전 종속, 테스트 불가, UX 함정 |
| **데이터** | JSON 전환 | MD: 파싱 불안정, UI 바인딩 불가, 테스트 어려움 |
| **v1 범위** | 팀원 관리 + 회의 실행 | 과도한 스코프 위험 |

## 6. 구현 계획 추천

### 결정 트리

```
타깃 사용자가 누구인가?
  │
  ├── 개발자 (Claude Code 사용자)
  │     └── Electron + React + TypeScript
  │         로컬 파일 접근 용이, 설치 경험 허용, 빠른 MVP
  │
  └── 비개발자 포함
        └── Next.js 웹앱 + API Route
            설치 없음, 접근성 최고, 배포 간단
```

**현실적 추천: Electron 먼저 (개발자 타깃 MVP)**

meeting-team의 지민(PD)이 가장 현실적인 조언을 했다: *"완성 가능한 스택"이 먼저다.* Electron은 JS/TS 개발자가 즉시 시작 가능하고, 로컬 파일 접근이 자연스럽다. 비개발자 확장은 MVP 검증 후 결정해도 늦지 않다.

---

### 추천 구현 스택

| 항목 | 결정 | 근거 |
|------|------|------|
| **프레임워크** | Electron | 개발자 타깃, 로컬 파일, 익숙한 생태계 |
| **UI** | React + TypeScript + Tailwind + shadcn/ui | 빠른 컴포넌트, 디자인 시스템 |
| **아키텍처** | Anthropic API 직접 호출 | 전 방식 합의 |
| **데이터** | JSON (profiles.json) | 전 방식 합의 |
| **상태 관리** | Zustand | 가볍고 단순 |
| **API 키** | OS 키체인 (keychain/credential store) | Electron 메인 프로세스 격리 |

---

### 개발 순서 (MVP 기준)

```
Phase 1 — 기반 (1주)
  ├── Electron + React 프로젝트 초기화
  ├── profiles.md → profiles.json 마이그레이션 스크립트
  └── Anthropic API 연동 + 키 관리 (OS 키체인)

Phase 2 — 팀원 관리 (1주)
  ├── TeamMemberCard 컴포넌트
  ├── 고용/해고/발령 인터랙션
  └── 팀원 추가/수정 폼

Phase 3 — 회의 실행 (2주)
  ├── 회의 방식 선택 UI (1종부터: meeting-team 권장)
  ├── Anthropic API 병렬 호출 + 스트리밍
  ├── 진행 상태 표시 (N/8 완료)
  └── 부분 실패 retry 로직

Phase 4 — 마무리 (1주)
  ├── 토큰 예측 계산기
  ├── 회의 결과 저장
  └── 패키징 (win/mac)
```

### 어느 회의 계획을 따를 것인가?

| 참고 문서 | 활용할 내용 |
|----------|-----------|
| `02-meeting-multi` | **v1 범위 확정** (팀원 관리 + 회의 실행 + 토큰 예측 + retry 로직 + 로딩 상태) |
| `03-meeting-agent` | **기술 결정 검증** (API 직접·JSON은 전원 합의 → 확정) |
| `04-meeting-team` | **다음 회의 아젠다** (6개 질문, 상태 다이어그램 선행, 타깃 사용자 확정) |
| `01-meeting` | 참고 없음 (편향 너무 심함, 데이터도 잘못됨) |

> **meeting-multi의 결론 + meeting-team의 질문·우선순위** 조합이 가장 실용적인 구현 계획이 된다.
