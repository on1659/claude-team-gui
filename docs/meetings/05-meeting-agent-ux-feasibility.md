# 팀 회의: "클로드를 모르는 사람도 에이전트를 쉽게 고용할 수 있는가?"

- **일시**: 2026-03-24
- **방식**: /meeting-agent (8인 병렬 독립 에이전트)
- **참석자**: 기획자x2, 개발자x2, QA x1, UI x1, UX x1, PD x1

---

## 회의 주제

> 이 프로젝트의 목적은 클로드를 잘 모르는 사람들도 에이전트를 쉽게 고용하고 볼 수 있게 만드는 것이다.
> 지금까지 작업된 것을 봤을 때 그게 가능할까?

## 역할별 상세 의견

### 기획자 #1 — 사용자 니즈 (5년차)

**핵심 의견**

"클로드를 잘 모르는 사람도 쉽게"라는 목표와 현재 구현 사이에 진입 장벽의 간극이 크다. 비주얼(픽셀아트 오피스)은 친근감 측면에서 훌륭한 방향이지만, 그 앞단 — API 키 설정, 용어 이해, 팀원 선택 기준 — 이 해소되지 않으면 사용자는 오피스 화면조차 보지 못하고 이탈한다. **90% 완성도는 기술 완성도이지, 사용자 경험 완성도가 아니다.**

**우려사항**

- [Critical] **온보딩 절벽**: API 키가 무엇인지도 모르는 사용자가 첫 단계에서 대부분 이탈
- [High] **선택 맥락 부재**: 8명 나열 후 선택 요구는 메뉴판 없는 식당에서 주문하라는 것과 같음
- [High] **전문용어 장벽**: "Quick/Deep", "멀티에이전트" 등은 도메인 내부 용어
- [Medium] **팀 구성 저장 불가**: 반복 사용자의 재방문 동기 상실
- [Medium] **회의 이력 UI 미흡**: 결과를 기억해주는 도구가 되어야 함

**제안**

1. 온보딩을 제품의 첫 번째 기능으로 설계 (3스텝 위저드)
2. 팀원 카드에 "사용 상황 힌트" 추가 (예: "화면 구성이 고민될 때")
3. 용어를 사용자 언어로 전면 교체 ("Quick" → "빠른 의견")
4. "내 단골 팀" 저장 기능을 우선순위 올리기

### 기획자 #2 — 비즈니스 전략 (8년차)

**핵심 의견**

현재 구현은 "클로드를 잘 아는 개발자"를 위한 도구로 완성도가 높다. 그러나 목표 대상인 "클로드를 잘 모르는 사람"에게는 진입장벽이 너무 높아 목적과 결과물 사이의 전략적 불일치가 존재한다. 90% 완성이라는 수치는 기술적 완성도이지, 시장 진입 준비도(Go-to-Market Readiness)가 아니다.

**우려사항**

- [치명] VS Code → Extension → API 키 발급 → 설정 → 사용 퍼널에서 비개발자 이탈률 90% 초과 예상
- [포지셔닝] 가치 약속("쉽게 고용")과 실제 경험(높은 기술적 비용) 간극이 브랜드 신뢰 훼손
- [시장 경쟁] ChatGPT, Claude.ai 등 제로-설정 AI 툴이 비개발자 시장 점유 중
- [수익화] API 비용이 사용자에게 전가되는 구조는 SaaS 모델에서 최대 이탈 요인

**제안**

- **단기**: 타겟을 "AI-savvy 개발자 및 테크 기획자"로 솔직하게 재정의
- **중기 (3~6개월)**: 웹 기반 진입점 추가 (zero-setup, OAuth, API 키 추상화)
- **핵심 원칙**: 에이전트 경험까지 5분 이내로 만드는 것이 최우선 과제

### 개발자 #1 — 백엔드 (6년차)

**핵심 의견**

아키텍처 자체는 충분히 구현 가능한 수준이다. postMessage 기반 통신은 VS Code 표준 패턴이고, 서비스 계층 분리도 적절하다. 다만 API 키 설정/관리 문제를 해결하지 않으면 첫 번째 장벽에서 막힌다.

**우려사항**

1. **API 키 관리**: 3개 LLM 키 각각 발급 필요. 저장 방식(SecretStorage vs 환경변수) 불명확
2. **비용 통제 부재**: 호출 전 예상 비용 확인 플로우 없음. 의도치 않은 과금 위험
3. **스트리밍 에러 핸들링**: 부분 실패 시 처리 불명확. 재시도/timeout/fallback 미검증
4. **team.json 직접 수정**: 비기술자에게 불가능한 작업
5. **Node.js 단일 스레드 병목**: 8개 동시 스트리밍 시 VS Code 느려질 가능성

**제안**

- 단기: API 키 설정 마법사 + 데모 모드, 비용 확인 다이얼로그, partial failure 전략
- 중기: team.json GUI 편집기, 비용 상한선 설정, AbortController 명시적 관리

### 개발자 #2 — 프론트엔드 (3년차)

**핵심 의견**

React 18 + TypeScript + Vite + 게임 엔진 구현은 기술적으로 탄탄하다. 그러나 "클로드를 잘 모르는 사람도 쉽게"라는 목표와 현재 UI 복잡도 사이에 명확한 긴장이 있다. 아이소메트릭 오피스 뷰는 인상적이지만, 처음 본 사용자에게 행위를 안내하는 것이 더 중요하다.

**우려사항**

- **온보딩 부재**: Quick/Deep 차이, 팀원 토글 이유를 즉각 이해하기 어려움
- **게임 엔진의 UX 역효과**: 맥락 이해 없으면 노이즈. 결과 UI로의 시선 이동 방해
- **스트리밍 UX 피로감**: 토큰 흐르는 화면을 처음 보면 "고장난 건가?" 반응 가능
- **VS Code 의존성**: 비개발자가 VS Code를 기본으로 쓰는지 불분명

**제안**

1. 첫 화면에 "무엇을 해주는 도구인지" 한 줄 + 샘플 주제 제안

3. 스트리밍에 단계 레이블 추가 ("분석 중", "토론 중", "합의 도출 중")

### QA — 품질보증 (5년차)

**핵심 의견**

"클로드를 잘 모르는 사람도 쉽게 쓸 수 있게"라는 목표와 현재 구현 사이에 검증되지 않은 진입 장벽이 다수 존재한다. 90% 완성도는 코드 기능 기준이며, **사용자 경험 기준으로는 50% 미만**으로 봐야 한다. 비개발자 대상 시나리오는 단 한 번도 테스트된 적이 없는 상태다.

**우려사항**

- [P0] **API 키 미설정 시 에러 처리 미검증**: 의미 있는 에러 메시지 없이 빈 화면/크래시 가능
- [P0] **8명 동시 호출 중 부분 실패**: 회의가 어중간하게 멈추는지, 실패 에이전트만 스킵하는지 불명
- [P0] **비개발자 전체 플로우 미테스트**: Extension 설치 → 첫 회의까지 Happy Path 미검증
- [P1] **네트워크 단절 시 스트리밍 복구**: 8개 동시 스트림 중 일부만 끊길 가능성
- [P1] **게임 엔진 + 8개 스트리밍 동시 렌더링 성능**: 저사양 PC 성능 기준치 미정의
- [P2] **다국어 미지원**: UI 텍스트 품질 일관성 미검증

**제안**

1. "완전 비개발자 시나리오" Walkthrough Test 최소 1회 실행
2. API 키 미설정 / 부분 실패 / 네트워크 단절 3대 에러 케이스 UI 메시지 확인
3. 8개 동시 스트리밍 중 2개 실패 시나리오 의도적 재현 테스트

### UI 디자이너 (4년차)

**핵심 의견**

현재 UI는 "에이전트를 쉽게 고용한다"는 목적보다 기술 시연에 가깝게 설계되어 있다. 아이소메트릭 오피스는 인상적이지만, 클로드를 모르는 사용자에게는 인지 부하를 먼저 준다. 사이드바 → 메인패널 전환 흐름이 단계적으로 느껴지지 않아, 첫 진입 사용자가 무엇을 해야 하는지 파악하기 어렵다.

**우려사항**

1. **온보딩 부재**: "에이전트 = 팀원?" 개념 설명 없이 폼만 노출
2. **아이소메트릭 씬 목적 불명확**: 사용자 태스크와 직접 연결되지 않음
3. **상태 색상 컨텍스트 부족**: 어떤 UI 요소에 적용되는지 불명확. 색각 이상 미고려
4. **컴포넌트 계층 과다**: 4단계 전환에 일관된 네비게이션 패턴 없음
5. **VS Code 테마 종속**: 비개발자에게 "독립 앱인가 플러그인인가" 혼란

**제안**

1. 첫 진입 온보딩 카드 삽입 (접을 수 있는 Disclosure 패턴)
2. 사이드바를 Stepper UI (1·2·3 스텝)로 재구성
3. 아이소메트릭 씬을 배경/최소화 위젯으로 전환
4. 상태 표현에 아이콘+라벨 병행 (색상만 의존하지 않도록)
5. ProgressBar를 전 화면 상단 고정으로 연속성 확보

### UX 디자이너 (7년차)

**핵심 의견**

현재 플로우는 "AI를 모르는 사람"이 아니라 "AI를 이미 아는 개발자"를 위해 설계되어 있다. API 키 입력이라는 첫 관문에서 클로드를 모르는 기획자/PM의 70%는 이탈할 것이다. "에이전트를 쉽게 고용한다"는 목적은 현재 플로우와 근본적으로 충돌한다.

**우려사항**

- **진입 장벽**: VS Code → Extension → API 키 3단계가 각각 별도 학습 비용
- **팀원 선택 의미 부재**: 판단 근거 없이 8명 나열. 프리셋 없어 매번 인지 부하 반복
- **Quick/Deep 맥락 부재**: 차이 모르면 "잘못 고르면?" 불안감
- **결과 이력 부재**: 기억되지 않는 도구는 쓸 이유 없음
- **아이소메트릭 연출**: 상태 피드백이 시각적 연출에 묻힐 수 있음

**제안**

1. API 키 전에 인터랙티브 데모/샘플 회의 재생 제공 — 첫 경험이 결과물이어야 함
2. 팀원 카드에 역할 설명 ("리스크를 먼저 찾아줍니다", "사용자 관점으로 반박합니다")
3. 상황별 프리셋 ("기획 초기", "아이디어 검증", "릴리즈 전 QA")
4. Quick/Deep을 시간 기준 표현 ("5분 / 핵심 의견만", "15분 / 충돌과 합의까지")
5. 결과 이력을 사이드바 기본 뷰로 배치

### PD — 프로젝트 디렉터 (10년차)

**핵심 의견**

기능 완성도(~90%)는 인상적이나, 목표와 플랫폼 선택 사이에 근본적인 모순이 있다. "클로드를 모르는 사람도 쉽게"라는 목표를 VS Code + API 키 구조에서 달성하기는 구조적으로 불가능하다. 기능은 완성되어 가지만 목표는 멀어지고 있다.

**우려사항**

1. **목표-플랫폼 불일치 (심각)**: 비개발자가 온보딩 과정을 독립 완수할 확률 낮음
2. **1인 개발의 범위 과잉 (높음)**: 미완성 항목이 전부 UX 관련 = 목표 달성의 핵심
3. **검증 부재 (중간)**: 실제 비개발자 대상 테스트 기록 없음

**제안 — 양자택일**

| 선택지 | 내용 | 의미 |
|--------|------|------|
| **A (권고)** | 타깃을 "AI 관심 개발자"로 좁힌다 | 현재 구조 그대로 완성. 빠른 출시 가능 |
| **B** | 목표 고수 → 플랫폼 변경 | 웹앱화/Claude 플러그인 검토. 상당 부분 재설계 |

1인 개발 리소스 기준, **선택지 A가 현실적**. 목표 문구를 현실에 맞게 조정하고 마무리하는 것이 가장 빠른 출시 경로.

## 공통 합의점

1. **기술 완성도는 높다** — 아키텍처, 게임 엔진, 스트리밍, 요약 등 핵심 기능 잘 구현됨
2. **온보딩이 최대 병목** — VS Code + API 키 진입 관문에서 비개발자 대부분 이탈
3. **"쉽게 고용" 메타포 실현** → 팀원 카드에 "이 사람이 뭘 해주는가" 설명 필수
4. **남은 10%가 목표 달성의 핵심** — 미완성 항목 전부 UX/온보딩 관련

## 주요 쟁점

| 쟁점 | A안 (현실론) | B안 (이상론) |
|------|-------------|-------------|
| 타겟 재정의 | "AI 관심 개발자"로 좁히고 빠른 출시 | 목표 고수 + 웹앱 진입점 추가 |
| 아이소메트릭 오피스 | 배경/위젯으로 격하, 결과 UI 전면 배치 | 핵심 차별화 자산 유지, 온보딩만 보강 |
| Quick/Deep 용어 | "빠른 의견 / 깊은 토론"으로 교체 | "5분/15분" 시간 기준 표현 |

## 권장 다음 액션

1. **타겟 사용자 결정 (즉시)**: "AI 관심 개발자" vs "비개발자 포함" — 모든 후속 작업의 방향 결정
2. **온보딩 마법사 구현 (최우선)**: API 키 설정 3단계 위저드 + 샘플 회의 데모 모드
3. **팀원 카드 역할 설명 추가 (저비용 고효과)**: 한 줄 역할 설명으로 선택 맥락 제공
