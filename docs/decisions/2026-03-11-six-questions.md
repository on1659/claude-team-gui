## Q1. 프로필 편집 UI가 초기 범위인가? (태준 → FE/UX)

> 편집 UI가 있으면 JSON 전환은 협상 대상이 아님. 데이터 설계 선행 조건.

**결정: v1 제외 — 파일 직접 수정**

- v1에서 팀원 추가/수정은 `meeting-team-profiles.md` 직접 편집
- GUI 편집 UI는 v2 범위
- **단, JSON 전환은 v1부터 필요** — 편집 UI가 없더라도 UI 바인딩·표시에 JSON이 필요

**영향**
- 데이터 모델: profiles.json으로 전환 (편집 UI 없이도)
- v1 범위가 줄어듦 → 일정 단축 가능

## Q2. CLI 미설치 환경 에러 핸들링 범위는? (현우 → 개발자)

> CLI 래핑 방식에서 Claude Code 미설치 환경 에러를 GUI에서 얼마나 숨길 수 있나?
> 이게 비개발자 타깃 가능 여부의 분기점.

**결정: 질문 자체가 무효 — CLI 의존성 없음**

4종 회의 전원 합의: **Anthropic API 직접 호출** 확정.
Claude Code CLI 설치 여부가 이 앱에 영향을 주지 않음.

**v1 에러 처리 범위 (API 기준)**
| 에러 | 처리 방식 |
|------|---------|
| API 키 없음 | 설정 화면으로 유도 |
| API 키 만료/잘못됨 | 인라인 에러 + 설정 링크 |
| 레이트 리밋 (429) | 재시도 대기 + 안내 |
| 타임아웃 (30초+) | 에이전트별 retry 1회 |
| 부분 실패 (8명 중 N명) | 실패 에이전트만 재시도 |

## Q3. 웹앱 기준 MVP 몇 주 소요? (소연 → 개발자)

> 출시 타이밍이 제일 중요한 변수. 구체적인 숫자 필요.

**결정: Electron 선택 — 웹앱 대비 빠름**

| 항목 | Electron | 웹앱 (Next.js) |
|------|---------|--------------|
| 로컬 파일 접근 | 기본 제공 | 별도 API 서버 필요 |
| API 키 보안 | OS 키체인 (메인 프로세스) | 서버 필요 or 노출 위험 |
| 배포 | 패키징 1회 | 호스팅 + 서버 유지 |
| 예상 MVP 기간 | **4~5주** | **5~6주** (서버 추가 구조 때문) |

→ Electron이 이 프로젝트 특성(로컬 파일, API 키 관리)에 더 맞고 오히려 빠름.

**Electron 기준 일정**
```
1주: Electron + React 초기화, profiles.json 전환
1주: 팀원 관리 UI (카드, 목록)
2주: 회의 실행 (API 호출, 병렬 에이전트, 스트리밍, retry)
1주: 마무리 (토큰 예측, 결과 저장, 패키징)
```

## Q4. 에이전트 동시 스트리밍이 가능한가? (승호 → 개발자)

> 동시 스트리밍이면 레이아웃과 포커스 관리 전략이 완전히 달라짐.

**결정: 가능 — 카드별 독립 스트리밍**

Anthropic API는 요청당 독립 스트림 지원.
8개 에이전트를 `Promise.all`로 병렬 실행하면 각자 스트리밍됨.

**레이아웃 전략**
```
[지민 ████████░░]  [현우 ██░░░░░░░░]
[소연 ██████████]  [태준 ████░░░░░░]
[미래 ░░░░░░░░░░]  [윤서 ████████░░]
[다은 ██░░░░░░░░]  [승호 ██████░░░░]

상단: 전체 진행률 "6/8 완료"
각 카드: 실시간 텍스트 스트리밍
완료된 카드: 결과 즉시 표시
```

**구현 방식**: 각 카드가 독립적으로 streaming state 보유.
전체 완료 후 요약 패널 표시.

## Q5. 모드 전환이 런타임인가, 회의 시작 시 선택인가? (다은 → 개발자)

> 동적 전환이면 레이아웃 전환 애니메이션·상태 관리까지 처음부터 설계에 포함 필요.

**결정: 회의 시작 시 선택 — 런타임 전환 없음**

- 회의를 시작할 때 방식 선택 (meeting / meeting-multi / meeting-agent / meeting-team)
- 회의 중에는 모드 변경 불가 (새 회의로 다시 시작)
- **앱 재시작은 불필요** — 이전 회의 종료 후 새 회의 시작 시 선택

**UI 설계**
```
[새 회의 시작]
  → 주제 입력
  → 방식 선택: ○ meeting  ○ meeting-multi  ● meeting-team  ○ meeting-agent
  → 참여 팀원 선택
  → [실행]
```

**영향**: 레이아웃 전환 애니메이션 불필요. 상태 관리 단순화.

## Q6. 회의 방식 3종 상태 다이어그램을 개발 전에 작성할 수 있나? (윤서 → 개발자)

> 상태 전이 조건·실패 시 롤백 시나리오가 문서화되어야 개발 시작 가능.

**결정: 필수 선행 작업 — 구현 전 작성**

**meeting-team 상태 다이어그램**
```
[대기] → [설정] → [실행 중] → [부분 완료] → [전체 완료] → [결과 표시]
                                    ↓
                              [에이전트 실패]
                                    ↓
                              [retry 시도] → 성공 → [부분 완료]
                                           → 실패 → [부분 완료로 마무리]
```

**각 상태 정의**
| 상태 | 조건 | 허용 액션 |
|------|------|---------|
| 대기 | 초기 | 주제 입력, 방식 선택, 팀원 선택 |
| 실행 중 | API 호출 시작 | 취소만 가능 |
| 부분 완료 | N/8 완료 | 대기 중 |
| 전체 완료 | 8/8 완료 또는 최종 retry 후 | 저장, 새 회의, 재시도 |
| 에러 | 개별 에이전트 실패 | retry (자동 1회) |

→ 이 다이어그램을 구현 시작 전 확정해야 QA 테스트 시나리오 작성 가능.
