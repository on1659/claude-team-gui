## 3. 방식별 상세 분석

### `/meeting-multi` — ~16K 토큰 / 16점

**무엇을 잘했나**

- 8역할이 자연스러운 토론 흐름으로 연결. 비용 극소
- 소연(전략)이 "최소 2가지 모드(비용 선택권)" 주장 → 비즈니스 관점의 날카로운 지적
- 태준(BE)이 메시지 프로토콜 타입 정의를 코드로 제시 → 즉시 활용 가능
- 승호(UX)가 "빠른 회의 / 심층 회의" 용어 번역 → 사용자 언어 전환
- 결론이 명확: Sidebar+Panel 분리, 회의 2종, fire-and-forget+seq

**무엇이 부족했나**

- 현우(기획)가 v1 범위를 먼저 정의하자 이후 역할들이 그 틀 안에서만 논의 → **편향 발생**
- "회의 2종"이라는 결론이 나왔지만, agent/team에서는 "1종 먼저 안정화"로 수렴 → **첫 발언이 결론을 유도**
- 스킬 파일 미주입이라 각 역할의 분석이 일반적 수준에 머무름
- AgentCard 상태가 4종(default/hover/inactive/disabled)으로 단순화됨 — agent/team에서는 6종으로 정교화

**적합한 상황**: 빠르게 방향 잡을 때. 토큰 아끼면서 8역할 의견이 필요할 때.

---

### `/meeting-agent` — ~329K 토큰 / 18점

**무엇을 잘했나**

- 편향 없는 독립 관점 — 8명이 서로의 의견을 모르고 작성
- 윤서(QA)의 리스크 등급이 가장 체계적: CRITICAL/HIGH/MEDIUM/LOW 4단계 + 테스트 시나리오 10개
- 다은(UI)이 CSS 변수 매핑 19개 완성 + 디자인 토큰 3계층 정의
- 승호(UX)가 낙관적 UI 3단계, 컨텍스트 회의(코드 우클릭→주제 자동 채움) 구체 설계
- 소연(전략)이 KPI "7일 재실행율 50%+"를 핵심 지표로 제시 → 다른 방식에서 나오지 않은 인사이트

**무엇이 부족했나**

- 토큰 20.6배 — 비용 부담 큼
- 다음 회의 아젠다가 될 "크로스 질문"이 없음 (서로의 존재를 모르니까)
- 충돌 지점이 표 정리 수준 — 누가 왜 다른지 맥락이 부족
- 지민(PD)이 Sonnet으로 실행됨 (team에서는 Opus) → PD의 판단 깊이가 상대적으로 얕음

**적합한 상황**: 중요한 기술 결정에서 편향 없는 분포 확인. 각 역할의 독립적 깊이가 필요할 때.

---

### `/meeting-team` — ~345.8K 토큰 / 22점

**무엇을 잘했나**

- **스킬 파일 기반 전문 분석** — 각 팀원이 자기 역할의 프레임워크(체크리스트, 리스크 매트릭스, DoD 등)에 맞춰 분석
- 지민(PD)이 Opus로 실행 → **"Phase 3이 일정의 38%"**, **"잘 되면 5일, 삐끗하면 8일"** 같은 경험 기반 판단
- **크로스 질문 8건 자연 발생** — "태준에게 묻고 싶은 것" 형태로 다음 회의 아젠다가 즉시 생성
  - "API RPM은?" (지민→태준)
  - "첫 청크 도달 시간?" (현우→태준)
  - "Panel race condition?" (미래→태준)
  - "AbortController 경로 검증?" (윤서→개발팀)
- 태준(BE)이 **공수를 항목별 세분화** (0.5일 단위, 총 7일) — agent에서는 "6일"이라고만 했음
- 다은(UI)이 **AgentCard 6상태 시각 스펙**을 VS Code 변수 기반으로 완성
- 현우(기획)가 **"8개 동시 스트리밍이 오히려 읽기 어려울 수 있다"** → CLI보다 못한 경험이 될 리스크 지적
- 소연(전략)이 **출시 채널 ROI 표** 제시 + GitHub Copilot Chat 경쟁 경고

**무엇이 부족했나**

- 가장 비쌈 (21.6x)
- 스킬 파일이 아직 Electron 맥락 → VS Code Extension으로 갱신 필요 (회의 중에도 지민이 지적)

**적합한 상황**: 아키텍처 결정 + 구현 계획 수립. 다음 회의 아젠다가 필요할 때. 가장 실전적.

## 4. 주요 쟁점 — 방식별 결론 비교

### v1 회의 모드 수

| 방식 | 결론 | 논거 |
| ---- | ---- | ---- |
| multi | **2종** | 소연: 비용 선택권 필수 (multi + team) |
| agent | **1종** | 현우/지민: 1종 먼저 안정화 |
| team | **1종** | 전원 수렴: meeting-team만으로 핵심 가치 검증 가능 |

> **교훈**: multi에서 소연의 "2종" 주장이 설득력 있었지만, 독립 사고(agent/team)에서는 "1종 안정화"가 우세. 순차 토론의 편향이 결론을 바꿀 수 있다.

### 메시지 프로토콜 정교도

| 방식 | 수준 | 상세 |
| ---- | ---- | ---- |
| multi | 기본 | fire-and-forget + seq만 |
| agent | 중간 | chunkIndex + fullContent |
| team | 완성 | chunkIndex + fullContent + ACK 패턴 + 에러 분류 + 공수 세분화 |

> **교훈**: 스킬 파일 주입이 프로토콜 설계의 깊이를 결정. multi에서 "충분하다"던 것이 team에서는 "부족하다"로 드러남.

### 비용 표시 방식

| 방식 | 결론 |
| ---- | ---- |
| multi | 실시간 표시 (암묵적) |
| agent | 회의 전 예상 + 완료 후 최종. 실행 중 미표시 |
| team | v1은 팀원 선택/해제 시 정적 계산만. 실시간은 v2 |

> **교훈**: 깊이 분석할수록 "실시간"의 정의가 불명확해지고, 스코프를 줄이는 방향으로 수렴.

## 5. 전 방식 공통 합의 (변하지 않는 결정)

3가지 방식 모두에서 반대 없이 동의한 항목:

| 항목 | 결정 | 근거 |
| ---- | ---- | ---- |
| **구조** | Sidebar Provider + Webview Panel 분리 | 목업 레이아웃이 VS Code 패턴과 정확히 매핑 |
| **통신** | Extension Host가 모든 API 호출 독점 | Webview는 렌더링 전용. 보안+상태 단일화 |
| **보안** | SecretStorage만. Webview 전달 금지 | API 키 노출 CRITICAL 등급 |
| **테마** | VS Code CSS 변수 전면 채택 | 하드코딩 HEX 제거, 자동 테마 연동 |
| **스트리밍** | 이중 디바운스 (Host 50ms + Webview 100ms) | 렌더링 부하 최소화 |
| **스코프** | 프로필 편집 UI 미포함 (v2) | 전 방식 합의 |
| **UI 피드백** | 낙관적 UI (버튼 즉시 → 카드 렌더 → 스트리밍) | 현우/승호 합의 |

## 6. 이전 회의(gui-구현방향)와의 차이

| 항목 | gui-구현방향 (03/10) | vscode-extension-ux (03/13) |
| ---- | ------------------- | --------------------------- |
| **프레임워크** | Electron vs 웹앱 vs Tauri 삼파전 | VS Code Extension **확정** |
| **데이터** | MD→JSON 전환 논의 | JSON 확정 + SecretStorage |
| **메시지** | IPC (Electron) | postMessage (VS Code Webview) |
| **API 키** | OS 키체인 (keytar) | VS Code SecretStorage |
| **스킬 파일** | Electron 맥락 | VS Code 맥락으로 갱신 중 |
| **쟁점 수준** | 프레임워크 선택 (근본 결정) | 구현 세부사항 (stagger 간격, ACK) |

> **진화**: 3일 전 회의에서 "무엇을 만들 것인가"를 결정했고, 이번 회의에서 "어떻게 만들 것인가"의 세부사항을 확정. 프레임워크 논쟁이 사라지고 기술 구현 깊이가 급격히 높아짐.

### 추천 구현 스택 (확정)
