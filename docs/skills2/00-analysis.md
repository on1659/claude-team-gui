## 1. Skills 2.0 핵심 요약

**한 줄 정의**: 스킬이 "참고 문서(매뉴얼)"에서 "프로그래밍 가능한 소프트웨어 모듈"로 진화

### 5대 핵심 기능

| # | 기능 | 설명 |
|---|------|------|
| 1 | Context Fork (격리 실행) | 스킬이 메인 컨텍스트를 소모하지 않고 별도 컨텍스트에서 실행 |
| 2 | 스킬별 훅 (Hooks) | 글로벌 훅 외에 스킬마다 독립적 훅 라이프사이클 보유 |
| 3 | Evals (품질 자동 검증) | AI가 스킬의 정상 동작을 정량적으로 자가 검증·보고 |
| 4 | Skill Creator (생성 자동화) | 스킬 생성→Evals YML 생성→핫 리로드→테스트까지 자동화 |
| 5 | Classification (수명 관리) | YAML 프론트 메타로 워크플로 분류, 모델 독립적 프로세스 관리 |

### 1.0 vs 2.0 비교

| 항목 | 1.0 | 2.0 |
|------|-----|-----|
| 본질 | 참고 문서 | 소프트웨어 모듈 |
| 컨텍스트 | 메인에서만 실행 | Context Fork로 격리 실행 |
| 수정 후 | 세션 재시작 필요 | 핫 리로드 (세션 유지) |
| 품질 검증 | 수동 테스트 | Evals 자동 검증 |
| 훅 | 글로벌 훅만 | 스킬별 독립 훅 |
| 생성 | 수동 작성 (30분~2시간) | Skill Creator (10~30분) |

## 2. 우리 프로젝트 현재 상태

### 스킬 파일 현황 (8개)

| 파일 | 역할 | 라인수 | YAML 프론트 메타 |
|------|------|--------|-----------------|
| skill-pd.md | PD 지민 | ~96 | ❌ 없음 |
| skill-planner-research.md | 리서처 현우 | ~62 | ❌ 없음 |
| skill-planner-strategy.md | 전략 소연 | ~79 | ❌ 없음 |
| skill-backend.md | BE 태준 | ~524 | ❌ 없음 |
| skill-frontend.md | FE 미래 | ~525 | ❌ 없음 |
| skill-qa.md | QA 윤서 | ~153 | ❌ 없음 |
| skill-ui.md | UI 다은 | ~625 | ❌ 없음 |
| skill-ux.md | UX 승호 | ~374 | ❌ 없음 |

### 현재 부재한 것들

- ❌ YAML Front Meta — 메타데이터 없이 순수 마크다운
- ❌ Context Fork — 모든 스킬이 메인 컨텍스트에서 실행
- ❌ 스킬별 훅 — `.claude/settings.json`에 기본 permissions만 존재
- ❌ Evals — 스킬 품질 검증 체계 없음
- ❌ Classification — 워크플로 분류 체계 없음
- ❌ CLAUDE.md — 프로젝트 루트 설정 파일 없음

## 3. GAP 분석 — 적용 가능 vs 불필요

### 적용해야 할 것 (HIGH)

#### 3-1. YAML Front Meta 추가
- **이유**: 8개 스킬 파일에 역할·분류·도구·에이전트 메타데이터가 전혀 없음
- **효과**: 스킬 자동 라우팅, 분류, 수명 관리 가능
- **난이도**: 낮음 (각 파일 상단에 YAML 블록만 추가)
- **우선순위**: ★★★

#### 3-2. Context Fork 적용
- **이유**: meeting-team 실행 시 8개 에이전트가 동시에 메인 컨텍스트를 소모
- **효과**: 각 에이전트 스킬이 격리된 컨텍스트에서 실행 → 컨텍스트 절약
- **난이도**: 중간 (프론트 메타에 `context: fork` 한 줄 추가)
- **우선순위**: ★★★

#### 3-3. Classification으로 워크플로 분류
- **이유**: 8개 스킬이 "회의 전용"인데 분류 체계가 없음
- **효과**: 회의 스킬 vs 개발 스킬 vs 분석 스킬 구분, 수명 관리
- **난이도**: 낮음
- **우선순위**: ★★☆

### 적용하면 좋은 것 (MEDIUM)

#### 3-4. 스킬별 훅 정의
- **이유**: 회의 시작/종료 시 추가 컨텍스트 주입 가능
- **효과**: 예를 들어 BE 태준 스킬 호출 시 → 자동으로 폴더 구조 정보 주입
- **난이도**: 중간~높음 (훅 스크립트 작성 필요)
- **우선순위**: ★★☆

#### 3-5. Additional Context 활용
- **이유**: 스킬 파일 자체가 500줄 이상인 경우 있음 (BE, FE, UI)
- **효과**: 핵심만 스킬에 두고, 상세 정보는 훅으로 필요 시에만 주입
- **난이도**: 중간
- **우선순위**: ★★☆

### 지금은 불필요 (LOW)

#### 3-6. Evals (스킬 품질 검증)
- **이유**: 우리 스킬은 회의 페르소나 주입 용도 → 자동 검증보다 회의 결과로 판단
- **효과**: 개인 스킬 개발 시 유용하나, 회의용 스킬엔 과도
- **우선순위**: ★☆☆

#### 3-7. Skill Creator (생성 자동화)
- **이유**: 8개 팀원 프로필이 이미 확정·작성됨, 추가 스킬 생성 계획 없음
- **우선순위**: ★☆☆

## 4. 핵심 인사이트 (영상에서)

> "AI 네이티브 개발에서 경쟁력은 모델 성능이 아니라 **워크플로(프로세스)**다."

> "스킬 2.0의 본질은 **방법론**이다. 사전이나 지식이 아니고 소프트웨어 방법론이 됐다."

> "비킷을 교보재처럼 이해한 다음, **각 조직에 맞게 워크플로를 구축**해야 한다."

우리 프로젝트에 적용할 때:
- 스킬 파일은 "참고 문서"가 아니라 **에이전트를 프로그래밍하는 모듈**로 취급
- 컨텍스트 효율 = 비용 절감 + 품질 향상 (Context Fork 필수)
- 스킬 메타데이터 = 자동 라우팅의 기반 (어떤 회의에 어떤 스킬을 불러올지)
