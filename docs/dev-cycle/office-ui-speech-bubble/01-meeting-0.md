### 팀원별 의견

**지민 (PD)** — v1 범위 초과 경고. 2단계 접근 제안: Phase A(말풍선+채팅로그, 1.5주) → Phase B(오피스뷰+애니메이션, 2주). 버그 수정은 이미 완료. 전체 UI 리디자인은 v1 M3 일정(4~5주)에 2주 추가 리스크. 스코프 크립 경보 발동하나 사장 지시이므로 조건부 GO.

**현우 (기획자① 리서치)** — 기존 카드뷰를 기본으로 유지하고 오피스뷰는 토글 옵션 제안. 레퍼런스: Gather.town, Discord Stage. 말풍선은 최대 2줄 truncate, 채팅로그에 전문 표시. 사용자 리서치 없이 전면 교체는 리스크.

**소연 (기획자② 전략)** — 전략적 차별화 요소. 즉시 개발 권장. 오피스뷰가 "AI 팀 회의" 컨셉의 핵심 비주얼. 경쟁 제품 대비 독창성 확보 가능. 카드뷰 병행은 불필요한 유지보수 비용.

**태준 (BE 개발자)** — 백엔드 변경 최소. 기존 메시지 프로토콜(agentDone, streamChunk 등) 재사용 가능. 말풍선용 chunk 요약 로직만 추가 필요(Extension Host에서 처리). 스트리밍 중 말풍선에 마지막 문장만 표시하는 방식 제안.

**미래 (FE 개발자)** — SVG+CSS 접근 선호. 예상 공수 ~20시간/3~4일. 주요 작업: OfficeView 컴포넌트, SpeechBubble 컴포넌트, ChatLogPanel 컴포넌트, PixelAvatar 애니메이션 연결. 레이아웃: 오피스뷰(flex-grow) + 채팅로그(280px 고정 우측). 반응형: 패널 폭 600px 미만 시 채팅로그 탭 전환.

**윤서 (QA)** — 5.5일 테스트 예상. 핵심 테스트: 말풍선 overflow/truncation, 8인 동시 스트리밍 시 말풍선 겹침, 채팅로그 스크롤 성능(1000+ 메시지), 반응형 전환 시 상태 유지, 기존 기능(회의 시작/종료/retry) 회귀 테스트.

**다은 (UI 디자이너)** — 5일 작업 예상. 말풍선 디자인 스펙: 최대 120px 너비, 꼬리 방향 캐릭터 중앙, VS Code 테마 연동(--vscode-editor-background 활용). 오피스 배경: 단색 or 미니멀 그리드. 캐릭터 배치: 2행 4열 그리드, 상태별 시각 피드백(idle=정지, streaming=talk 애니메이션, done=이펙트).

**승호 (UX 디자이너)** — 레이아웃 비율 40:60(오피스:채팅) 제안. 사용자 주의 분산 방지를 위해 말풍선은 현재 발화자만 표시하고 나머지는 fade. 채팅로그에 역할별 아이콘+색상 구분. 키보드 접근성: Tab으로 캐릭터 간 이동, Enter로 상세 보기.

### 주요 합의점

1. **SVG+CSS 접근** — Canvas가 아닌 SVG+CSS로 오피스뷰 구현 (접근성, 테마 호환, 기존 PixelAvatar 재사용)
2. **말풍선 truncate** — 말풍선에는 1~2줄 요약만, 전문은 채팅로그에 표시
3. **기존 프로토콜 재사용** — agentReducer, 메시지 프로토콜 대부분 재사용 가능
4. **CSS 애니메이션 연결** — component.css에 이미 정의된 talk/spin/blink 키프레임을 PixelAvatar에 연결 (quick win)
5. **반응형 대응** — 좁은 패널에서는 채팅로그를 탭/토글 전환

### 주요 충돌 지점

| 주제 | 찬성 | 반대 | 쟁점 |
|------|------|------|------|
| 카드뷰 유지 여부 | 현우(토글) | 소연(제거) | 유지보수 비용 vs 사용자 선택권 |
| v1 범위 포함 여부 | 소연(즉시) | 지민(v2) | 일정 리스크 vs 전략적 차별화 |
| 오피스:채팅 비율 | 승호(40:60) | 미래(flex:280px) | 고정비율 vs 유연한 레이아웃 |

- **현우 → 승호**: "카드뷰↔오피스뷰 토글 시 UX 전환 비용이 얼마나 되나?"
- **소연 → 지민**: "v1에서 오피스뷰 제외하면 제품 차별화 요소가 뭔가?"

- **미래 → 다은**: "말풍선 최대 너비 120px이면 한글 기준 몇 글자? 줄바꿈 정책은?"
- **윤서 → 미래**: "8인 동시 스트리밍 시 말풍선 업데이트 주기는? rAF 기반 배칭 적용되나?"
- **다은 → 승호**: "다크/라이트 테마 전환 시 말풍선 색상 자동 대응 범위는?"
- **승호 → 현우**: "사용자 테스트 없이 오피스뷰 전면 적용해도 되나? A/B 테스트 고려?"

### 기술 의존성

```
다은(UI 스펙) ──→ 승호(UX 레이아웃) ──→ 미래(FE 구현) ──→ 윤서(QA 테스트)
                                          ↑
태준(BE 요약 로직) ───────────────────────┘
```

- **블로킹 경로**: 다은 UI 스펙 → 미래 FE 구현 (가장 긴 의존 체인)
- **병렬 가능**: 미래의 PixelAvatar 애니메이션 연결은 스펙 없이 독립 진행 가능
- **병렬 가능**: 태준의 메시지 프로토콜 확장은 미래 인터페이스 확정 후 병렬 진행
