## [최신] 직군별 전문 스킬 파일 생성

**논의 흐름**
`/meeting-team`이 페르소나(이름, 성격, 말투)만 주입하고 있어서, 분석 품질의 한계가 있었다.
"스킬을 넣어야 한다"는 힌트 → "직군별로 스킬 파일을 만들 수 있는가?" 질문으로 이어짐.

**결정 사항**
역할별 전문 방법론을 담은 스킬 파일을 `.claude/skills/`에 생성하고,
`meeting-team` 커맨드가 프로필과 함께 스킬 파일을 읽어 각 에이전트 프롬프트에 주입하도록 업데이트.

**생성 파일**
```
.claude/skills/
├── README.md
├── skill-pd.md                 ← 지민 (PD): Go/No-Go, 리스크 매트릭스, ICE 점수, 스코프 크립
├── skill-planner-research.md   ← 현우 (기획자①): 사용자 스토리, JTBD, MoSCoW, 검증 방법
├── skill-planner-strategy.md   ← 소연 (기획자②): 비즈니스 임팩트, KPI, 경쟁 포지셔닝, 로드맵
├── skill-backend.md            ← 태준 (개발자 BE): API 설계, OWASP 보안, 성능 기준, 데이터 모델
├── skill-frontend.md           ← 미래 (개발자 FE): 컴포넌트 설계, 상태 관리, Electron IPC
├── skill-qa.md                 ← 윤서 (QA): 테스트 피라미드, 엣지케이스, DoD, Anthropic API 특수 케이스
├── skill-ui.md                 ← 다은 (UI): 디자인 시스템, 시각적 계층, 빈/로딩/에러 상태
└── skill-ux.md                 ← 승호 (UX): 사용자 여정, WCAG 접근성, 인터랙션 원칙
```

**업데이트 파일**
- `.claude/commands/meeting-team.md` — 스킬 파일 로드 + 에이전트 프롬프트 주입 로직 추가

**기대 효과**
- 이전: 페르소나만 있음 → 말투는 다르지만 분석 깊이 비슷
- 이후: 페르소나 + 전문 방법론 → 역할별 체크리스트·프레임워크 실제 적용
- 토큰 증가: 스킬 파일 분량만큼 약 3~5천 토큰 증가 예상

## 에이전트 품질 개선 방법 리서치

**배경**
4종 회의 비교 결과(05-meeting-result.md) 작성 후, `/meeting-team`이 21/25로 가장 높은 점수를 받았지만
"페르소나만 주입된 상태"라는 한계가 있었음. "어떻게 더 잘할 수 있을까?" 논의 시작.

**리서치 결과 (claude-code-guide 에이전트 활용)**

에이전트 품질을 높이는 방법:
1. **도구 접근권 부여** — Read, WebSearch 등 도구를 에이전트가 사용하게 함
2. **프로젝트 컨텍스트 주입** — 프로젝트 문서를 프롬프트에 포함
3. **역할별 스킬(도메인 지식) 주입** — 전문 프레임워크·체크리스트 포함
4. **구조화된 출력 형식** — 역할에 맞는 아웃풋 포맷 명시

사용자 힌트: "스킬을 넣어야 한다" → 역할별 스킬 파일 아이디어로 구체화됨.

## 4종 회의 비교 결과 분석 및 문서화

**주제**: GUI 구현 방향 결정을 위한 4가지 회의 방식 비교

**실측 토큰 사용량**
| 방식 | 에이전트 수 | 토큰 | 비율 |
|------|-----------|------|------|
| `/meeting` | 1개 | 9,286 | 1x |
| `/meeting-multi` | 1개 | 12,109 | 1.3x |
| `/meeting-agent` | 8개 병렬 | 81,434 | 8.8x |
| `/meeting-team` | 8개 병렬 | 85,165 | 9.2x |

**점수 결과**
| 방식 | 점수 (25점 만점) |
|------|---------------|
| `/meeting` | 13점 |
| `/meeting-multi` | 17점 |
| `/meeting-agent` | 20점 |
| `/meeting-team` | 21점 |

**핵심 발견: 편향 문제**
- `meeting` / `meeting-multi`: 순차 실행 → 앞 발언이 뒤 역할에 영향 → Electron으로 수렴
- `meeting-agent` / `meeting-team`: 독립 실행 → 실제로는 웹앱 지지가 더 많거나 팽팽

**전 방식 공통 합의 (확정 사항)**
- 아키텍처: Anthropic API 직접 호출 (CLI 래핑 불안정)
- 데이터: JSON 전환 (MD 파싱 불안정, UI 바인딩 불가)
- v1 범위: 팀원 관리 + 회의 실행

**구현 계획 추천**
- 스택: Electron + React + TypeScript + Tailwind + shadcn/ui + Zustand
- 참고 문서: multi(v1 범위 확정) + team(다음 아젠다 질문 6개)
- 생성 문서: `docs/meetings/gui-구현방향/05-meeting-result.md`
